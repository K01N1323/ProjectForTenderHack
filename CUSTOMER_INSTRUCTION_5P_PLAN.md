# План дальнейших действий по инструкции заказчика

## 1. Что важно из инструкции заказчика

Из инструкции по работе с Порталом для заказчика видно, что реальный пользовательский сценарий для заказчика выглядит так:

1. Заказчик заходит в каталог СТЕ и ищет нужную продукцию.
2. Переходит в реестр СТЕ, отфильтрованный по категории.
3. Открывает карточку СТЕ.
4. На карточке видит:
   - общие сведения;
   - предложения;
   - характеристики;
   - классификацию.
5. На карточке СТЕ ищет и фильтрует оферты.
6. Добавляет подходящую оферту в корзину.
7. Из корзины создаёт один из типов закупки:
   - прямая закупка;
   - котировочная сессия;
   - закупка по потребности.
8. Для прямой закупки выбирает оферты, указывает количество, этапы поставки, получателя и место поставки, затем направляет поставщику.

Ключевые места из инструкции, на которые нужно опираться в продукте:

1. Каталог СТЕ и поиск продукции: страницы `98-103`.
2. Поиск оферт внутри карточки СТЕ: страницы `103-105`.
3. Корзина и создание закупки: страницы `105-109`.
4. Сценарий прямой закупки: страницы `110-114`.
5. Права заказчика на корзину, прямую закупку и КС: страницы `14-15`.
6. Переходы через меню и разделы портала: страницы `17-18`, `66`.

## 2. Что это меняет для вашего решения

Это означает, что ваш хакатонный прототип нельзя делать как абстрактный поиск по товарам.
Он должен быть встроен в реальный заказческий путь:

1. `умный поиск СТЕ`
2. `карточка СТЕ`
3. `поиск и ранжирование оферт`
4. `добавление в корзину`
5. `создание закупки`

Поэтому лучший MVP для защиты:

1. Пользователь вводит запрос по СТЕ.
2. Система показывает релевантные СТЕ с учётом опечаток, синонимов и истории закупок.
3. Пользователь открывает карточку СТЕ.
4. Система показывает оферты и ранжирует их.
5. Пользователь добавляет оферту в корзину.
6. Пользователь создаёт прямую закупку или хотя бы доходит до выбора типа закупки.
7. Система объясняет, почему выбранная СТЕ или оферта оказалась выше.

## 3. Что делаем дальше

Ниже не общий план, а план старта, который можно запускать прямо сейчас.

### Этап A. Зафиксировать границы MVP

Нужно зафиксировать, что вы показываете на защите:

1. Поиск по СТЕ с исправлением опечаток.
2. Персонализированную выдачу СТЕ.
3. Карточку СТЕ с основными вкладками.
4. Ранжирование оферт внутри карточки.
5. Добавление в корзину.
6. Выбор типа закупки.
7. Минимальный flow прямой закупки.

Что можно не делать полноценно:

1. Полную регистрацию и авторизацию.
2. Полную реализацию всех видов закупок.
3. Полную контрактную часть.
4. Полную интеграцию с ЭП, ЕИС и реальными внешними системами.

### Этап B. Зафиксировать продуктовую декомпозицию

Разбиваем решение на 5 параллельных треков:

1. Данные и поисковый индекс.
2. Персонализация и ранжирование.
3. Backend и API.
4. Frontend и пользовательский сценарий.
5. Аналитика, BPMN, demo-flow и защита.

## 4. Распараллеливание на 5 человек

### Человек 1. Search/Data Engineer

Зона ответственности:

1. Индекс СТЕ.
2. Нормализация запросов.
3. Поисковый baseline.
4. Опечатки и синонимы.

Что делать прямо сейчас:

1. Взять `data/processed/ste_catalog_search_ready.csv`.
2. Собрать baseline-поиск по полям:
   - `clean_name`;
   - `normalized_name`;
   - `category`;
   - `normalized_category`;
   - `key_tokens`.
3. Подготовить `20-30` тестовых запросов:
   - точное совпадение;
   - словоформы;
   - опечатки;
   - синонимы.
4. Сделать функцию `search_ste(query, top_k)`.
5. Подготовить словарь частотных синонимов.
6. Подготовить модуль исправления опечаток.

Результат к концу первого блока:

1. Есть `top-20` СТЕ на любой запрос.
2. Есть демонстрация, что опечатка и словоформа не ломают поиск.

### Человек 2. Ranking/Personalization Engineer

Зона ответственности:

1. Персонализация СТЕ.
2. Персонализация оферт.
3. Session-based ранжирование.
4. Объяснимость.

Что делать прямо сейчас:

1. Взять:
   - `customer_ste_stats.csv`;
   - `customer_category_stats.csv`;
   - `region_category_stats.csv`.
2. Собрать профиль заказчика:
   - топ-категории;
   - часто покупаемые СТЕ;
   - региональные предпочтения.
3. Определить формулу бустинга:
   - category affinity;
   - history affinity;
   - region affinity;
   - session action boost.
4. Сделать функцию `rerank_ste(results, customer_profile, session_state)`.
5. Подготовить функцию объяснения:
   - "часто закупалось этой организацией";
   - "похоже на ранее выбранные категории";
   - "популярно в регионе";
   - "поднято после клика/корзины".

Результат к концу первого блока:

1. Есть слой персонализации поверх базового поиска.
2. Есть заметный сдвиг результатов на подготовленных кейсах.

### Человек 3. Backend Engineer

Зона ответственности:

1. API.
2. Сервисные модели.
3. Session state.
4. Корзина и mock flow закупки.

Что делать прямо сейчас:

1. Поднять `FastAPI`.
2. Сделать endpoints:
   - `GET /search/ste?q=...`
   - `GET /ste/{ste_id}`
   - `GET /ste/{ste_id}/offers`
   - `POST /cart/add`
   - `GET /cart`
   - `POST /cart/create-procurement`
   - `POST /session/event`
3. Подготовить DTO для:
   - карточки СТЕ;
   - результата поиска;
   - оферты;
   - корзины;
   - объяснения ранжирования.
4. Если оферт в данных нет в явном виде, сделать mock-offers на основе:
   - истории контрактов по СТЕ;
   - поставщиков;
   - регионов;
   - ценовых агрегатов.
5. Сделать моковый сценарий прямой закупки:
   - выбрать позиции;
   - указать количество;
   - выбрать тип закупки;
   - сформировать объект заявки.

Результат к концу первого блока:

1. Фронтенд уже может работать поверх живого API.
2. Полный demo-flow можно пройти без ручных костылей.

### Человек 4. Frontend Engineer

Зона ответственности:

1. UI поиска.
2. UI карточки СТЕ.
3. UI оферт.
4. UI корзины и создания закупки.

Что делать прямо сейчас:

1. Собрать 4 экрана:
   - экран поиска СТЕ;
   - экран карточки СТЕ;
   - экран корзины;
   - экран выбора типа закупки.
2. На экране поиска показать:
   - строку поиска;
   - исправленный запрос;
   - список СТЕ;
   - причины ранжирования.
3. На карточке СТЕ показать:
   - общие сведения;
   - характеристики;
   - список оферт;
   - фильтры оферт;
   - кнопки действий.
4. На корзине показать:
   - выбранные оферты;
   - итоговую сумму;
   - выбор позиций;
   - кнопку создания закупки.
5. Сделать явный эффект персонализации:
   - до действия;
   - после клика;
   - после добавления в корзину.

Результат к концу первого блока:

1. Есть визуально понятный пользовательский сценарий.
2. Есть что показывать жюри уже до полной полировки.

### Человек 5. Product/Analyst/Presentation Lead

Зона ответственности:

1. Синхронизация команды.
2. BPMN.
3. Ручные кейсы.
4. Метрики.
5. Презентация и защита.

Что делать прямо сейчас:

1. На основе инструкции заказчика зафиксировать основной пользовательский путь:
   - поиск СТЕ;
   - выбор СТЕ;
   - поиск оферты;
   - корзина;
   - создание закупки.
2. Собрать `10-15` реальных поисковых сценариев для ручной оценки.
3. Подготовить матрицу метрик:
   - Recall@10;
   - MRR@10;
   - nDCG@10;
   - доля кейсов с полезным изменением после персонализации.
4. Нарисовать BPMN не по вашей внутренней архитектуре, а по заказческому процессу.
5. Подготовить структуру презентации и demo script.
6. Жёстко вести трекер статуса каждые `30-45` минут.

Результат к концу первого блока:

1. У команды есть единый demo script.
2. У вас не разваливается защита даже если разработка ещё не закончена.

## 5. Пошаговый план на ближайшие 4 часа

### Первые 30 минут

1. Все читают этот план и фиксируют зоны ответственности.
2. Человек 5 создаёт общий трекер:
   - `todo`;
   - `in progress`;
   - `blocked`;
   - `done`.
3. Человек 3 создаёт каркас backend.
4. Человек 4 создаёт каркас frontend.
5. Человек 1 готовит baseline-поиск.
6. Человек 2 готовит профиль пользователя и формулу rerank.

### 30-90 минут

1. Человек 1 доводит поиск СТЕ до рабочего baseline.
2. Человек 2 доводит профиль заказчика и session boosts.
3. Человек 3 собирает API на поиске и карточке СТЕ.
4. Человек 4 подключает UI поиска к API.
5. Человек 5 готовит список кейсов и сценарий демо.

Контрольная точка:

1. Запрос вводится.
2. Список СТЕ возвращается.
3. В UI уже есть список результатов.

### 90-150 минут

1. Человек 1 добавляет опечатки и синонимы.
2. Человек 2 добавляет объяснения ранжирования.
3. Человек 3 делает API оферт и корзины.
4. Человек 4 делает карточку СТЕ и корзину.
5. Человек 5 формирует BPMN и ручной benchmark.

Контрольная точка:

1. Пользователь открывает карточку СТЕ.
2. Видит оферты.
3. Может добавить оферту в корзину.

### 150-240 минут

1. Человек 1 и 2 вместе калибруют качество выдачи.
2. Человек 3 делает mock flow создания закупки.
3. Человек 4 доделывает экран выбора типа закупки.
4. Человек 5 собирает презентацию и репетирует story.

Контрольная точка:

1. Есть полный путь от поиска до корзины.
2. Есть выбор типа закупки.
3. Есть объяснение, почему выбранный результат находится выше.

## 7. Главные зависимости между людьми

1. Человек 4 зависит от Человека 3 по API-контракту.
2. Человек 3 зависит от Человека 1 по поисковому модулю.
3. Человек 3 зависит от Человека 2 по rerank и explanation payload.
4. Человек 5 зависит от всех, но не должен ждать конца разработки, а должен собирать материал параллельно.

Чтобы не словить простой:

1. Человек 3 сразу задаёт стабильные mock-ответы.
2. Человек 4 сначала работает на моках.
3. Человек 1 и 2 потом просто подменяют логику под тот же контракт.

## 8. Что нельзя провалить

1. Нельзя делать только умный поиск без сценария корзины и закупки.
2. Нельзя делать только красивый интерфейс без реального изменения выдачи.
3. Нельзя делать персонализацию, которую невозможно заметить на демо.
4. Нельзя откладывать презентацию и BPMN на конец.

## 9. Самый сильный demo-flow

1. Заказчик вводит запрос с опечаткой.
2. Система исправляет запрос и показывает релевантные СТЕ.
3. Пользователь видит объяснение, почему конкретная СТЕ поднялась выше.
4. Пользователь открывает карточку СТЕ.
5. Находит и выбирает оферту.
6. Добавляет оферту в корзину.
7. Переходит к созданию закупки.
8. Система показывает, что путь соответствует реальной логике Портала для заказчика.

## 10. Вывод

После чтения инструкции заказчика правильный фокус такой:

1. Вы делаете не просто "поиск по каталогу".
2. Вы делаете интеллектуальный вход в реальный процесс закупки заказчика.
3. Успешный прототип должен начинаться в поиске СТЕ и заканчиваться хотя бы выбором типа закупки или черновиком прямой закупки.

Именно это нужно брать как основу всей дальнейшей разработки и всей защиты.
